Overview

This project extends the OptiX path tracer built through Homeworks 1–4 — which already has GGX BRDFs, next-event estimation, and multiple importance sampling — with volumetric light transport for heterogeneous participating media. The goal was a photorealistic CS2-style smoke grenade screen. However simply the smoke rendering proved to be challenging initself.

The full pipeline was designed and implemented: custom AABB geometry in OptiX, a delta-tracking integrator with Henyey-Greenstein phase sampling, volume NEE with transmittance shadow rays, and a .rawvdb density loader. The VDB import broke at the CMake/NanoVDB linkage stage; the renders below use a procedurally generated density field as a fallback. That was later replaced using Houdini shell python that is noted within scene / final this is executed to export following vdb to rawvdb. Implementation

  • Ray–AABB intersection — slab method registered as an OptiX custom primitive; flags the hit as a volume entry for the integrator.
  • Trilinear density lookup — maps world position into [0,1]³ within the AABB and performs 8-corner trilinear interpolation over the voxel grid.
  • Delta tracking (Woodcock) — free-path sampling under a scalar majorant σt,maj; real vs. null collision decided per-step against local density. Unbiased through heterogeneous media.
  • Henyey-Greenstein phase function — analytic importance sampling via inversion formula; single parameter g controls forward/backward scattering anisotropy.
  • Volume NEE — at each real collision, a shadow ray is traced to every quad light and weighted by phase function value and geometry term.
  • Russian roulette + indirect scatter — throughput modulated by single-scattering albedo σs/σt; surviving paths launch an indirect scatter ray, depth-limited to 8 events.
  • .rawvdb loader — reads a binary header (nx, ny, nz) and a flat float32 voxel array exported from Houdini via a Python script, normalises to [0,1], and uploads to an OptiX buffer. The Houdini export script is in the testing folder. NanoVDB CMake linkage fails on the lab machines, so only rawVDB data was loaded for the final renders.

What the images show

Both renders use a procedural 3D noise field as a stand-in for real VDB data. The dark cloud-like regions are high-density clusters where σt is large enough to absorb most incident light. The contrast render is more technically telling — the darker patch on the rear wall and floor is real volumetric attenuation: shadow rays from those surface points pass through the medium and are weighted by transmittance, reducing the NEE contribution.
